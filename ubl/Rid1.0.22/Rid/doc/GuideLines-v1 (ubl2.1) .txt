namespaces:
http://schemas.rfsholding.com/[schemaname]/[MessageName]/v[version.subversion];
example:
http:/schemas.rfsholding.com/rid/SalesOrder/v1.0

filename:
[SchemaName]-[MessageName]-v[version].[subversion].xsd
example:
Rid-SalesOrder-v1.0.xsd

The Ubl schemas are the basic schemas where the Rid schemas are copied from. The Rid contain a subset of the Ubl schemas in such a way that the result is Ubl compliant.

So elements can be deleted from a message or type when it is optional.
Cardinality can be narrowed down in the following way:
0..n => 1..n
0..n => 1..1
0..n => m..n
1..n => 1..1
1..n => m..n
0..1 => 1


When a message does not exist a new message can be created, using the Ubl 2.1 common basic & aggregate components in the original .


Extra aggregate or basic elements can be created, to be used in new message types or in the extension section, in a seperate set of extension schemas. These elements should be constructed from the Ubl common elements where possible. The custom elements are stored in:
- Rid-ExtendedCommonBasicComponents-v1.0.xsd for basic elements
- Rid-ExtendedCommonAggregatedComponents-v1.0.xsd for aggregated elements

Each xsd uses the utf-8 codepage.

Schema files that are not message specific should be positioned in the Rid/common directory.
In order to avoid dependancy between different messages,we consider the aggregate components as 'message-specific' . 

Message specific are: 
	[SchemaName]-[MessageName]-v[version].[subversion].xsd 
	[SchemaName]-[MessageName]AggregateComponents-v[version].[subversion].xsd.
	and all the files related to exentensions.
All message specific files are grouped in a subset.

xmlns:udt="http://schemas.rfsholding.com/rid/UnqualifiedDataTypes/v1.0"
